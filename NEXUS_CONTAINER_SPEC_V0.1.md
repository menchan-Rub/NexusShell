# NexusContainer: 次世代コンテナシステム - 仕様書・技術要件 (v0.1 Draft)

**1. はじめに**

**1.1. プロジェクト概要**

NexusContainerは、セキュリティ、パフォーマンス、カスタマイズ性、そして開発者体験において既存のコンテナ技術（特にDocker）を超えることを目指す、次世代のコンテナランタイムおよび管理ツールです。Linuxカーネルの最新機能を活用し、軽量かつ堅牢なコンテナ分離環境を提供します。NexusShellプロジェクトからスピンオフし、汎用的なコンテナソリューションとしての利用も想定しています。

**1.2. 設計目標**

*   **卓越したセキュリティ:** 最小権限の原則を徹底し、デフォルトでセキュアな環境を提供。きめ細かいセキュリティポリシー設定を可能にする。
*   **軽量性とパフォーマンス:** 高速なコンテナ起動、低オーバーヘッドな実行時パフォーマンスを実現。
*   **高度なカスタマイズ性:** コンテナの隔離レベル、リソース割り当て、ネットワーク構成などを柔軟に設定可能にする。
*   **開発者フレンドリー:** 直感的で使いやすいCLIとAPIを提供し、コンテナの構築・実行・管理を容易にする。
*   **ポータビリティと互換性:** OCI (Open Container Initiative) 標準への準拠を最大限考慮し、既存のエコシステムとの連携も視野に入れる。
*   **NexusShellとの親和性:** NexusShellの実行環境として最適化されつつ、スタンドアロンツールとしても機能する。

**1.3. 対象ユーザー**

*   セキュリティとパフォーマンスを重視する開発者・運用者。
*   特定のワークロードに最適化されたコンテナ環境を求めるユーザー。
*   コンテナ技術の内部構造を理解し、カスタマイズしたい上級ユーザー。
*   NexusShellユーザー。

**2. アーキテクチャ**

**2.1. 全体構成**

NexusContainerは、以下の主要コンポーネントで構成されます。

*   **`nexusd` (デーモンプロセス - Optional):**
    *   コンテナのライフサイクル管理（作成、起動、停止、破棄）。
    *   イメージ管理（プル、プッシュ、保存）。
    *   ネットワーク管理。
    *   ストレージ管理（ボリューム、レイヤー）。
    *   APIエンドポイントの提供 (gRPC/REST)。
    *   将来的にはデーモンレス運用も可能な設計を目指す。
*   **`nexus-ctl` (CLIクライアント):**
    *   `nexusd` と対話し、コンテナとイメージを管理するためのユーザーインターフェース。
    *   Dockerライクな直感的なコマンド体系を目指す。
*   **`nexus-runtime` (低レベルランタイム):**
    *   OCIランタイム仕様に準拠（またはそれに近いインターフェースを提供）。
    *   実際にLinuxカーネル機能（名前空間, cgroups, etc.）を操作し、コンテナプロセスを起動・管理する。
    *   `nexusd` から呼び出される。
*   **`libnexuscontainer` (コアライブラリ):**
    *   コンテナ操作のコアロジックを実装するライブラリ。
    *   `nexusd`, `nexus-ctl`, `nexus-runtime` の共通基盤。
    *   将来的には他言語バインディングを提供し、外部ツールからの利用も可能にする。

**2.2. 隔離技術**

*   **名前空間:**
    *   PID, Network, Mount, UTS, IPC, User名前空間をフルサポート。
    *   User名前空間を利用したルートレスコンテナをデフォルトで推奨。
*   **コントロールグループ (cgroups v2):**
    *   CPU, メモリ, I/O, PIDなどの詳細なリソース制限とアカウンティング。
    *   Systemdとの連携も考慮。
*   **ケーパビリティ:**
    *   コンテナプロセスに付与するケーパビリティをきめ細かく制御。
    *   デフォルトで最小限のケーパビリティセットを適用。
*   **Seccomp-BPF:**
    *   許可/拒否するシステムコールを詳細に定義できるフィルタリング機構。
    *   アーキテクチャ依存の考慮。
    *   事前定義されたプロファイルとカスタムプロファイルのサポート。
*   **任意アクセス制御 (MAC):**
    *   AppArmor / SELinux プロファイルの適用をサポート (オプション)。
*   **ファイルシステム:**
    *   OverlayFS (または同等のレイヤードファイルシステム技術) を利用したイメージレイヤーとコンテナの書き込み可能レイヤー。
    *   ルートファイルシステムの堅牢な隔離 (`pivot_root`)。
    *   ボリュームマウント、tmpfsマウント。

**2.3. イメージ仕様**

*   OCI Image Format Specification への準拠を基本方針とする。
    *   既存のDockerイメージとの互換性を最大限確保。
    *   独自のメタデータ拡張も検討。
*   イメージビルド機能 (`nexus-ctl build`):
    *   Dockerfileライクな定義ファイルからのイメージ構築。
    *   ビルドキャッシュ、マルチステージビルド。

**2.4. ネットワーク仕様**

*   CNI (Container Network Interface) プラグインモデルのサポートを検討。
    *   ブリッジ、ホスト、オーバーレイネットワークなど多様なネットワーク構成に対応。
*   デフォルトでコンテナごとのネットワーク名前空間を提供。
*   ポートマッピング。
*   内部DNS解決。

**2.5. ストレージ仕様**

*   名前付きボリューム、匿名ボリュームのサポート。
*   コンテナのストレージドライバ (OverlayFSなど)。
*   ボリュームプラグイン機構の検討。

**3. セキュリティ要件**

*   **デフォルトでセキュア:**
    *   ルートレスコンテナの推奨。
    *   最小限のケーパビリティ。
    *   厳格なSeccompプロファイルのデフォルト適用。
    *   読み取り専用ルートファイルシステムの推奨。
*   **ポリシーエンジン:**
    *   現在の `SandboxPolicy` を発展させ、より表現力豊かで管理しやすいポリシー定義言語を設計。
    *   ポリシーの継承、オーバーライド。
    *   ポリシー違反時の監査ログ。
*   **脆弱性管理:**
    *   イメージの脆弱性スキャン機能の統合 (Trivy, Clairなどの外部ツール連携または独自実装)。
*   **署名と検証:**
    *   イメージのデジタル署名と検証機能 (Notary v2 / Sigstore 互換など)。

**4. パフォーマンス要件**

*   **コンテナ起動時間:** 数百ミリ秒以内を目指す (アプリケーションによる)。
*   **ランタイムオーバーヘッド:** CPU、メモリオーバーヘッドを最小限に抑える。
*   **イメージ操作:** イメージのプル、プッシュ、ビルドの高速化。

**5. APIおよびCLI**

*   **API:**
    *   gRPCを主要なAPIプロトコルとして採用。RESTful APIゲートウェイも提供。
    *   明確に定義されたAPIスキーマ (Protobuf)。
    *   バージョニングと後方互換性。
*   **CLI (`nexus-ctl`):**
    *   Docker CLIのユーザー体験を参考にしつつ、より一貫性のあるコマンド体系。
    *   豊富なサブコマンドとオプション。
    *   スクリプティングに適した出力形式 (JSONなど)。

**6. 技術スタック (初期案)**

*   **言語:** Rust (パフォーマンス、安全性、低レベル制御のため)
*   **非同期ランタイム:** Tokio
*   **APIフレームワーク:** Tonic (gRPC), Actix-web/Axum (REST)
*   **LinuxカーネルAPI操作:** `nix`, `libc` クレートなど。
*   **cgroups操作:** `cgroups-rs` クレートまたは直接APIアクセス。
*   **ファイルシステム:** `overlayfs-rs` (もしあれば)、または直接システムコール。

**7. 開発ロードマップ (超初期案)**

*   **フェーズ0: 基盤技術の確立 (現在の `container.rs` の進化系)**
    *   `libnexuscontainer` のコア部分として、`child_fn` を中心としたLinuxネイティブサンドボックス機能の完成度向上 (pivot_root, ユーザー名前空間, cgroups v2, seccomp, capabilities)。
    *   この段階ではスタンドアロンツールではなく、ライブラリとしての機能充実に注力。
    *   NexusShellからの直接利用を想定。
*   **フェーズ1: ミニマルランタイムとCLI (`nexus-runtime`, `nexus-ctl` の原型)**
    *   OCIランタイムに近いインターフェースを持つ `nexus-runtime` のプロトタイプ開発。
    *   基本的なコンテナ起動・停止・削除を行える `nexus-ctl` の最小限実装。
    *   単純なディレクトリベースのイメージサポート。
*   **フェーズ2: イメージとネットワークの基礎 (`nexusd` の原型)**
    *   OverlayFSを利用した基本的なイメージレイヤー機能。
    *   `nexusd` の初期バージョンとして、ローカルでのイメージ管理とコンテナ管理機能。
    *   基本的なブリッジネットワーク。
*   **フェーズ3: APIとエコシステム連携**
    *   `nexusd` のgRPC APIの実装。
    *   OCIイメージ仕様への準拠強化。
    *   `nexus-ctl build` の基礎実装。
*   **フェーズ4以降: 機能拡充と最適化**
    *   ボリューム管理、高度なネットワーク機能、セキュリティ機能強化、パフォーマンスチューニング、ドキュメント整備、コミュニティ形成など。

**8. 懸念事項とリスク**

*   **開発リソース:** 広範囲な機能を高品質で実装するには、相当な開発期間と専門知識を持つチームが必要。
*   **セキュリティの担保:** 自前実装部分の脆弱性混入リスク。徹底的なテストと第三者レビューが不可欠。
*   **既存エコシステムとの競争:** Dockerという巨大なエコシステムと競合し、ユーザーを獲得することの難しさ。明確な差別化戦略が必要。
*   **Linuxカーネルへの深い依存:** カーネルバージョン間の互換性維持、新機能への追従。
